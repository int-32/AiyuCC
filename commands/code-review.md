# /code-review - 代码审查

使用 code-reviewer agent 审查代码变更，覆盖代码质量、安全漏洞、API 设计和数据库设计。

## 用户参数

$ARGUMENTS

支持的参数:
- 无参数: 审查所有未提交的变更
- 文件路径: 审查指定文件
- PR 编号: 审查指定 PR 的变更
- `--api`: 仅审查 API 设计
- `--db`: 仅审查数据库设计
- `--full`: 执行全量审查（代码 + API + 数据库）

## 执行步骤

### 1. 变更识别

- 运行 `git diff` 和 `git diff --cached` 查看所有变更
- 分类变更文件: 路由/控制器、模型/迁移、服务层、工具函数、测试等
- 识别涉及 API 端点和数据库模型的变更

### 2. 代码质量审查

使用 code-reviewer agent 逐文件审查:
- 安全性（注入、认证绕过、敏感数据泄露）
- 正确性（逻辑错误、边界条件、异常处理）
- 可维护性（命名、复杂度、重复代码）
- 检查相关测试是否覆盖变更

### 3. API 设计审计

当变更涉及 API 端点（路由、控制器、序列化器）时，审查以下维度:

| 审查维度 | 检查项 |
|----------|--------|
| **RESTful 规范** | URL 命名是否使用名词复数、HTTP 方法是否语义正确、是否滥用 POST |
| **版本控制** | 是否有 API 版本前缀（如 `/v1/`）、破坏性变更是否考虑兼容 |
| **请求设计** | 参数命名一致性、必填/可选标注、分页/过滤/排序是否标准化 |
| **响应设计** | 响应结构是否统一（code/message/data）、错误码是否规范、是否返回多余字段 |
| **安全性** | 认证/授权是否到位、是否有越权风险、敏感字段是否脱敏 |
| **性能** | 是否有 N+1 查询、大列表是否分页、是否缺少缓存策略 |
| **幂等性** | POST/PUT 操作是否幂等、重试是否安全 |
| **一致性** | 与项目现有 API 风格是否统一（命名、结构、错误处理） |

### 4. 数据库设计审计

当变更涉及数据库模型、迁移文件或 Schema 时，审查以下维度:

| 审查维度 | 检查项 |
|----------|--------|
| **表设计** | 命名是否规范（复数/单数一致）、是否有合适的主键策略 |
| **字段设计** | 类型选择是否合理（如 varchar 长度、decimal 精度）、是否有不必要的可空字段 |
| **索引** | 高频查询字段是否加索引、联合索引顺序是否合理、是否有冗余索引 |
| **外键与关联** | 关联关系是否正确（1:1/1:N/M:N）、级联删除策略是否合理、是否有孤儿数据风险 |
| **范式与冗余** | 是否满足 3NF、有意的反范式是否有充分理由、是否有不合理的数据冗余 |
| **迁移安全** | 迁移是否可逆、是否会锁表、大表变更是否考虑分批执行 |
| **扩展性** | 是否预留了扩展字段策略（如 JSON 字段 vs 新增列）、分表分库是否提前考虑 |
| **审计字段** | 是否有 created_at/updated_at、软删除是否按需使用 |

### 5. 输出审查报告

按严重性排序输出所有发现:

```markdown
## 审查报告

### CRITICAL（必须修复）
- [C1] (API) POST /api/users 缺少认证中间件，存在未授权访问风险
  - 建议: 添加 @auth_required 装饰器或路由级中间件

### HIGH（强烈建议修复）
- [H1] (DB) users 表 email 字段缺少唯一索引，可能导致重复注册
  - 建议: 添加 UNIQUE 约束 `ALTER TABLE users ADD UNIQUE (email);`
- [H2] (API) GET /api/orders 未实现分页，数据量大时会导致性能问题
  - 建议: 添加 limit/offset 或 cursor 分页，默认 page_size=20

### MEDIUM（建议修复）
- [M1] (DB) order_items 表与 orders 表的级联删除策略为 CASCADE，可能误删历史数据
  - 建议: 改为 SET NULL 或软删除，保留审计追溯能力
- [M2] (Code) 函数 process_payment 超过 80 行，建议拆分

### LOW（改进建议）
- [L1] (API) 响应中 created_at 格式不统一，部分返回 timestamp，部分返回 ISO 8601
  - 建议: 统一使用 ISO 8601 格式

### 总结
- 发现问题: X 个 (CRITICAL: a, HIGH: b, MEDIUM: c, LOW: d)
- API 审计: X 个端点已审查
- 数据库审计: X 个模型/迁移已审查
- 测试覆盖: 变更中 X% 有对应测试
```

## 调用 Agent

使用 Task 工具调用 code-reviewer agent:
- subagent_type: CodeReview
- 传入变更文件列表和 diff 内容
- 如涉及数据库，额外调用 database-reviewer agent

## 原则

- 每个问题必须给出**具体的修复建议或代码示例**，不能只说"不合理"
- API 审计以项目现有风格为基准，避免强推与项目不一致的规范
- 数据库审计要考虑**实际数据量和查询模式**，不做脱离场景的优化建议
- 区分"必须修复"和"建议改进"，不要把所有问题都标为 CRITICAL
