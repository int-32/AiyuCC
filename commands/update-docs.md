# /update-docs - 文档更新

检查并更新与代码变更相关的文档，重点维护后端项目的 README 体系。

## 用户参数

$ARGUMENTS

支持:
- 无参数: 自动检测项目结构，更新所有需要更新的文档
- 文件路径: 更新指定文档
- `--full`: 全量扫描并重建所有 README

## 执行步骤

### 1. 项目结构扫描

- 识别项目根目录和后端代码的入口目录
- 扫描一级子目录，识别各业务模块文件夹（排除 `node_modules`、`__pycache__`、`.git`、`dist`、`build` 等非业务目录）
- 检查根目录和每个模块目录下是否存在 `README.md`（大小写不敏感: `readme.md`/`README.md`/`Readme.md`）

### 2. 代码变更分析

- 运行 `git diff --name-only` 查看最近变更的文件
- 识别变更涉及的模块，确定需要更新哪些 README
- 如果使用 `--full` 参数，则对所有模块进行全量分析

### 3. 模块级 README 更新

对每个有变更（或 `--full` 模式下所有）的模块目录，检查并更新其 `README.md`:

**如果 README 已存在** — 阅读现有内容，在保留结构的基础上更新变更部分
**如果 README 不存在** — 跳过，不自动创建（提示用户哪些模块缺少 README）

模块 README 应包含以下章节（已有的保留并更新，缺失的补充）:

```markdown
# [模块名]

## 功能概述
[模块的核心职责和业务功能描述]

## 目录结构
[模块内的文件/子目录组织说明]

## API 接口

| 方法 | 路径 | 说明 | 请求参数 | 响应 |
|------|------|------|----------|------|
| POST | /api/xxx | 描述 | {...} | {...} |

## 数据库结构

### [表名/模型名]
| 字段 | 类型 | 说明 | 约束 |
|------|------|------|------|
| id | integer | 主键 | PK, auto |

[表关系说明]

## 数据流

[描述数据在模块内的流转路径，使用 Mermaid sequence/flowchart]

## 依赖关系
- 依赖的其他模块及用途
- 被哪些模块依赖

## 变更记录
| 日期 | 变更内容 | 关联 Commit |
|------|---------|------------|
| YYYY-MM-DD | 描述 | hash |
```

### 4. 项目根目录 README 更新

检查并更新项目根目录的 `README.md`，维护整体后端视角:

**如果 README 已存在** — 阅读现有内容，更新以下章节
**如果 README 不存在** — 跳过，提示用户

根目录 README 应包含以下章节:

```markdown
# [项目名]

## 项目概述
[项目背景、目标和核心功能]

## 技术栈
[使用的框架、语言、数据库等]

## 项目结构

| 目录 | 功能 | 说明 |
|------|------|------|
| module-a/ | 用户管理 | 用户注册、认证、权限 |
| module-b/ | 订单处理 | 订单创建、支付、状态流转 |

## 各模块核心功能

### [模块名]
- 核心功能点列表
- 关键 API 端点摘要

## 业务流程图

[使用 Mermaid 绘制核心业务流程]

## 数据流程图

[使用 Mermaid 绘制跨模块的数据流转关系]

## 快速开始
[环境搭建、依赖安装、启动命令]
```

### 5. 其他文档更新

检查以下文档是否需要同步更新:
- API 文档（如有独立的 OpenAPI/Swagger 文件）
- 配置文档（新增配置项）
- CHANGELOG.md（版本变更记录）

### 6. 输出报告

汇总本次文档更新:
- 更新了哪些 README（根目录 + 模块）
- 哪些模块缺少 README（提示用户是否需要创建）
- 变更摘要

## 原则

- **只更新已有的 README**，不自动创建新文件，缺失的仅提示
- **保留用户自定义内容** — 更新时保留现有结构和用户手写的段落，只修改与代码变更相关的部分
- **接口文档要具体** — API 接口写到方法/路径/参数级别，数据库写到字段/类型/约束级别
- **Mermaid 图表** — 业务流程和数据流优先使用 Mermaid，便于版本管理
- **变更记录追溯** — 模块 README 的变更记录要关联 git commit hash
- 不创建多余的文档文件
